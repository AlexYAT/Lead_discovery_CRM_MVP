# Summary

Операционный план **следующего** цикла после закрытия [DOA-OP-003](./DOA-OP-003.md) и финального snapshot [DOA-IMP-014](../implementation_snapshot/DOA-IMP-014.md): **операционная приёмка параллельной работы двух инстансов** приложения с общим файлом SQLite.

Цикл **не** продолжает выполнение задач OP-003 (WAL, lifecycle, busy_timeout, retry, короткие транзакции уже зафиксированы в IMP-011–013 и приняты в [DOA-VAL-002](../validation_reports/DOA-VAL-002.md)). Он адресует **остаточный разрыв** между:

- ожиданиями [DOA-IDEA-003](../idea_to_mvp/DOA-IDEA-003.md) (Definition of Done: устойчивость при **двух** параллельных инстансах, отсутствие `database is locked` в нормальном usage) и исходной формулировкой T05 в OP-003;
- фактическим scope [DOA-VAL-002](../validation_reports/DOA-VAL-002.md): валидация T01–T04 выполнена на **однопоточном** smoke, нагрузочное / многопроцессное доказательство **не** входило в scope.

**Решение, зафиксированное в этом OP:** следующий прагматичный шаг после завершения hardening по коду — **acceptance / verification hardening**: воспроизводимая проверка двух инстансов и оформление результатов в validation-артефакте; изменения кода допускаются только в отдельном будущем исполнении цикла и только если приёмка выявит дефект в пределах согласованного scope (не часть настоящего документа-создания).

# Goal

- получить **воспроизводимую** процедуру и фактические результаты работы **двух** экземпляров приложения (uvicorn) против **одной** БД MVP;
- прогнать согласованный набор CRUD/write сценариев (lead, contact, consultation) при параллельном доступе в рамках MVP;
- зафиксировать наблюдения (в т.ч. отсутствие или наличие transient lock-ошибок, целостность данных);
- явно сопоставить результат с ограничениями, описанными в DOA-VAL-002 и DOA-IMP-014.

# Scope of Execution

## Включено

- подготовка **playbook** (пошаговая процедура: порты, порядок старта двух инстансов, общий `DB_PATH`, сценарии действий);
- выполнение приёмочного прогона по playbook (человеком или с помощью минимальных скриптов/команд из репозитория — без обязательности автоматизации CI);
- сопоставление сценариев с уже используемым smoke-набором из DOA-VAL-002 (создание/обновление lead, contact attempt, consultation);
- оформление **нового** validation report ([DOA-VAL-003](../validation_reports/DOA-VAL-003.md) — ожидаемый номер на момент планирования);
- при отсутствии изменений коду — финальный **implementation snapshot** цикла приёмки ([DOA-IMP-015](../implementation_snapshot/DOA-IMP-015.md) — ожидаемый), фиксирующий «только верификация, код не менялся», либо при необходимости — точечные IMP после отдельных решений (вне create-only этого OP-документа).

## Не включено (out of scope)

- повторная реализация SQLite hardening (замкнуто в OP-003 / IMP-014);
- новые продуктовые функции, изменения UI/UX;
- переход на PostgreSQL, multi-user auth, распределённый lock;
- stress-тестирование с числом инстансов **больше двух**;
- обязательная разработка полноценного observability-стека (метрики APM и т.п.) — в рамках данного OP не планируется;
- правка исторических документов (create-only для новых артефактов исполнения).

# Стабилизированное состояние (вход в цикл)

На основе [DOA-IMP-014](../implementation_snapshot/DOA-IMP-014.md) и [DOA-VAL-002](../validation_reports/DOA-VAL-002.md) зафиксировано:

- WAL, short-lived connections, централизованный `busy_timeout`, write-only retry, explicit short transactions на целевом write path — **внедрены и согласованы**;
- продуктовая логика и публичные контракты сервисов в рамках hardening **не менялись**;
- **не доказано** поведение при двух параллельных инстансах в объёме VAL-002.

# Execution Strategy (T01–T04)

## T01 — Playbook двухэкземплярной приёмки

- описать воспроизводимые шаги: два процесса `uvicorn`, разные порты, одна БД (`data/lead_discovery_crm_mvp.db`);
- перечислить CRUD/write действия и порядок (в т.ч. пересечение операций по одному lead);
- определить критерии «pass/fail with limitations» (наличие `database is locked` после исчерпания retry, потеря записей и т.д.).

**Deliverable:** зафиксированный playbook (новый doc в `docs/` или расширение только через **новый** артефакт исполнения — не правка DOA-OP-004 метаданных без процедуры DocOps).

## T02 — Выполнение прогона

- выполнить приёмку строго по playbook T01;
- сохранить наблюдения (логи консоли, порядок операций, идентификаторы сущностей).

**Deliverable:** сырые результаты прогона, пригодные для включения в validation report.

## T03 — Validation report

- создать [DOA-VAL-003](../validation_reports/DOA-VAL-003.md) (ожидаемый ID) со статусом по фактическим результатам;
- явно связать выводы с DOA-VAL-002 known limitations и Definition of Done из DOA-IDEA-003.

**Deliverable:** принятый или draft validation report — по фактической практике проекта после исполнения.

## T04 — Закрытие цикла приёмки

- при отсутствии изменений коду: snapshot [DOA-IMP-015](../implementation_snapshot/DOA-IMP-015.md) с резюме цикла OP-004;
- при изменениях коду в рамках исполнения: отдельные IMP по правилам DocOps, затем итоговый snapshot цикла.

**Deliverable:** IMP snapshot, закрывающий OP-004.

# Definition of Done for execution

Цикл **DOA-OP-004** считается выполненным, если:

- playbook двухэкземплярной приёмки существует и воспроизводим;
- прогон по playbook выполнен и описан;
- validation report создан и согласован с фактом (принятие ограничений или подтверждение целей IDEA-003 в части двух инстансов — в зависимости от результата);
- итоговый implementation snapshot цикла OP-004 создан;
- не выполнялось расширение scope за пределы настоящего operational plan без нового DocOps-цикла.

# Risks and Controls

- **Риск:** flaky поведение при ручном прогоне. **Контроль:** фиксировать порты, версию Python, повторяемость шагов в playbook.
- **Риск:** ложное обобщение с двух инстансов на произвольную нагрузку. **Контроль:** в VAL явно указать ограничение «только два инстанса».
- **Риск:** смешение с OP-003. **Контроль:** любые изменения коду только с новым обоснованием и отдельными IMP; hardening-код не менять «по инерции».
