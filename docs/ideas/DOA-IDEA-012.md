## Title

Dotenv-based Local Config Loading

---

## Problem

Текущее приложение опирается на **переменные окружения процесса**, но в **локальном** runtime файл **`.env`** **не** подхватывается автоматически во всех операционных сценариях (CLI discovery, агентские/IDE subprocess, отдельная интерактивная PowerShell и т.д.).

Это приводит к проблемам:

- **API keys** и другие параметры из **`.env`** **не всегда** попадают в процесс приложения без ручного **`$env:...` / `export`**  
- поведение **зависит** от конкретной **shell-сессии** и способа запуска  
- усложняется **воспроизводимая** проверка live-интеграций (в т.ч. search)  
- локальная эксплуатация становится **менее предсказуемой** для оператора  

**Environment contract** и **config discipline** уже зафиксированы в DocOps (**IMP-033** и связанный цикл), но **использование** `.env` в типичном локальном runtime остаётся **не полностью прозрачным**: оператор должен помнить про ручную загрузку окружения.

---

## Idea

Добавить **централизованную** поддержку загрузки **`.env`** в **локальном** runtime через существующий **Config Layer** (ранняя инициализация до чтения `DiscoveryConfig`), с сохранением **единого входа** к настройкам для discovery.

**Цель:**

- сделать локальную подачу **API keys** и прочих **env-параметров** **воспроизводимой** при обычном запуске из репозитория  
- **сохранить** единый вход через **config layer** (не размножать чтение секретов по модулям)  
- **снизить** зависимость от ручного экспорта переменных в shell для **типового** локального сценария  
- **не** менять бизнес-логику **pipeline** discovery  

*(Отдельный DocOps-цикл; не является «исправлением задним числом» цикла Brave Search integration и не заменяет решения предыдущих baseline без ARCH/DEC.)*

---

## Scope

**В рамках IDEA (намерение, без выбора библиотеки и деталей реализации в этом документе):**

- **централизованная** загрузка **`.env`** для **локального** runtime (корень репозитория, согласованный путь) **до** или при первом обращении к **config layer** для discovery  
- использование **существующего** **Config Layer** как **единственной** точки доступа к уже описанным настройкам после загрузки  
- улучшение **воспроизводимости** локальных запусков и проверок интеграций для оператора и CI/dev окружений  

**Вне scope:**

- **redesign** всей конфигурационной системы приложения  
- **secret management** уровня production, **cloud secret stores**, vault  
- изменения **UI**  
- изменения **бизнес-логики** discovery pipeline (порядок стадий, семантика qualification и т.д.)  
- переработка **существующего** env contract **сверх** минимальной поддержки **локальной** загрузки `.env` (имена переменных и правила DEC/ARCH переносятся в следующий цикл при необходимости)
