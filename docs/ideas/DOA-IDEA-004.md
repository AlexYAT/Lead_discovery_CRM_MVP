# Summary

Следующий слой ценности MVP после стабильного **operational baseline** ([DOA-IMP-015](../implementation_snapshot/DOA-IMP-015.md)): сместить фокус с «учёт лида вручную» на **обнаружение и отбор** потенциальных лидов до попадания в CRM, с сохранением простого single-user сценария.

---

## Problem

Пользователь уже может **вести** лида в системе (статус, контакты, консультации), но **источник** лида по-прежнему внешний по отношению к продукту: профиль и контекст вносятся вручную. Реальная боль следующего уровня — **где брать кандидатов**, **как не утонуть в шуме**, **как быстро понять «стоит ли тратить время»** до создания записи в CRM. Без этого слоя MVP остаётся «журналом учёта», а не инструментом **lead discovery**.

---

## Context

Текущий MVP уже умеет:

- хранить лиды и ключевые поля профиля;
- переводить лиды по FSM статусов;
- фиксировать попытки контакта и консультации;
- показывать агрегаты на dashboard;
- работать в **multi-instance** режиме с одной SQLite БД ([DOA-IMP-015](../implementation_snapshot/DOA-IMP-015.md)).

Технический риск concurrent SQLite для MVP **снят** в заявленном объёме; можно безопасно наращивать **product layer**, не смешивая его с повторным hardening.

---

## Opportunity

Добавить ценность **до** карточки лида: воспроизводимый мини-процесс **поиска / отбора / импорта** кандидатов (sourcing) и лёгкое **обогащение контекста** (минимальные сигналы: тема, канал, ссылка, заметка оператора), чтобы пользователь тратил меньше ручного копипаста и быстрее принимал решение «взять в работу / отсечь».

---

## Possible directions

1. **Курируемый inbound + очередь на разбор**  
   Фиксированный канал (например CSV/ручной paste списка URL или никнеймов) → нормализация в «черновики» → пользователь отмечает «создать лид в CRM» для выбранных строк. Минимум интеграций, максимум контроля.

2. **Поисковые сессии с сохранением запроса**  
   Пользователь задаёт параметры поиска (платформа, ключевые слова, регион — в пределах MVP) → система сохраняет **сессию** и список кандидатов как read-only черновик → из строки одним действием создаётся lead с предзаполнением полей. Требует чёткого правового/ToS контекста для любых внешних API.

3. **Лёгкое enrichment при создании лида**  
   Расширение текущей формы создания лида: подсказки по домену/URL (заголовок страницы, описание — если разрешено политикой), без отдельной «очереди отбора». Меньший scope, быстрее в ценность «меньше ручного ввода».

---

## Recommendation

**Рекомендуется направление 1 — «Курируемый inbound + очередь на разбор».**

**Обоснование:** оно наименее зависит от нестабильных внешних API и юридически серых зон скрейпинга на первом шаге, при этом прямо отвечает на вопрос «откуда лиды»: пользователь **подтягивает** кандидатов известным ему способом (файл, буфер, список ссылок), а CRM даёт **место принятия решения** и связку с уже существующим workflow статусов. Направления 2 и 3 остаются развитием после появления ARCH/DEC по выбранным ограничениям (источники данных, лимиты MVP).

---

## Scope hint

Потенциально в MVP следующего этапа (на уровне идеи, без архитектуры):

- сущность или экран **«очередь кандидатов / черновики»**, отделённая от `leads` до явного «принять в CRM»;
- импорт из **одного** простого формата (например CSV с колонками platform / profile_name / profile_url) + ручная правка перед массовым или одиночным переносом;
- действие **«создать lead»** из черновика с переносом полей в текущую модель;
- явный **out of scope** на этом этапе: полноценный скрейпинг без согласованных источников, multi-user, сложные ML-скоринги.

Следующий DocOps-шаг после принятия идеи: **ARCH** (источники данных, границы MVP, модель черновика) и **DEC** — без реализации в рамках настоящего документа.
