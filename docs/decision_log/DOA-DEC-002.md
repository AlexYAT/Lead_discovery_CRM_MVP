# Summary
Документ закрепляет правила validation-cycle перед переходом к этапу OP.

# Context
Решения приняты на основании:
- `DOA-IDEA-002`;
- `DOA-ARCH-002`;
- подтверждённого decision checkpoint.

# Decision 1 — Validation Result Storage
## Варианты
- один документ на весь validation-cycle;
- один документ на одну validation-сессию.

## Принятое решение
Один документ = одна validation-сессия.

## Обоснование
- лучше соответствует create-only;
- удобнее трассируемость;
- проще сопоставлять с конкретным прогоном.

# Decision 2 — Problem Recording Granularity
## Варианты
- строгий шаблон;
- свободный текст.

## Принятое решение
Свободный текст.

## Обоснование
- на текущем этапе шаблон ещё не зафиксирован;
- снижает трение при первичной фиксации проблем;
- подходит для exploratory validation.

# Decision 3 — Severity Usage
## Варианты
- без severity;
- с severity low / medium / high.

## Принятое решение
Использовать severity low / medium / high.

## Обоснование
- помогает приоритизировать findings;
- упрощает следующий DEC/OP цикл;
- не усложняет validation критически.

# Decision 4 — Scenario Coverage Model
## Варианты
- свободный проход;
- обязательный чеклист сценариев.

## Принятое решение
Обязательный чеклист сценариев.

## Обоснование
- снижает риск неполного покрытия;
- соответствует ARCH validation-cycle;
- делает validation воспроизводимым.

# Decision 5 — Human-Run Readiness Before Validation
## Варианты
- начинать validation без формализованного запуска;
- сначала подготовить воспроизводимое окружение и запуск человеком.

## Принятое решение
Validation допускается только после подготовки окружения, установки зависимостей и явного способа
запуска человеком.

## Обоснование
- validation должен выполняться реальным человеком, а не только как концепция;
- без воспроизводимого запуска validation-cycle неполон;
- это создаёт основание для OP с environment/setup/run checklist.

# Consequences
- validation становится воспроизводимым;
- результаты будут храниться по сессиям;
- фиксация проблем остаётся гибкой;
- severity добавляет приоритизацию;
- следующий OP должен включать:
  - подготовку окружения;
  - установку зависимостей;
  - инструкции запуска;
  - чеклист сценариев validation.

# Constraints
- никаких улучшений системы до фиксации проблем;
- никаких решений в validation reports;
- никаких функциональных доработок в рамках validation-cycle DEC;
- только подготовка возможности запуска и выполнения validation.

# Next Steps
Следующий этап: `OP`.

OP должен декомпозировать:
1. подготовку окружения;
2. установку зависимостей;
3. воспроизводимый запуск человеком;
4. validation checklist;
5. формат session-based фиксации проблем.
