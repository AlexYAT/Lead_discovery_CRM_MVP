## Назначение

Воспроизводимая процедура для человека: два независимых процесса `uvicorn`, **один** файл SQLite, чередование HTTP-действий по SSR-формам приложения. Исполнитель фиксирует числовые идентификаторы по ходу сценария (без «угадывания» id заранее).

---

## Environment

| Параметр | Значение |
|----------|----------|
| **ОС** | Windows 10+ или иная с доступным Python 3.12+ |
| **Python** | **3.12+** (как в [RUN.md](../../RUN.md); минимум — то, что поддерживает зависимости проекта) |
| **Корень репозитория** | каталог, где лежит `app/`, `data/`, `requirements.txt` (ниже `<REPO>`) |
| **Рабочий каталог для всех команд** | `<REPO>` |
| **Путь к БД (единый файл)** | `<REPO>\data\lead_discovery_crm_mvp.db` (код: `app.db.database.DB_PATH` → `PROJECT_ROOT / "data" / "lead_discovery_crm_mvp.db"`) |
| **Запуск приложения** | `python -m uvicorn app.main:app --host 127.0.0.1 --port <PORT>` из активированного venv (см. [RUN.md](../../RUN.md)) |

Оба инстанса читают и пишут **в один и тот же** `DB_PATH`, потому что `PROJECT_ROOT` вычисляется от расположения `database.py` и не зависит от порта uvicorn.

## Setup

1. Открыть **два** терминала (окна PowerShell). В обоих:

   ```powershell
   Set-Location "<REPO>"
   .\.venv\Scripts\Activate.ps1
   ```

   Если venv ещё не создан — выполнить шаги 1–3 из [RUN.md](../../RUN.md), затем вернуться сюда.

2. **Инициализация схемы и проверка файла БД** (выполнить **один раз** в любом из терминалов, пока **ни один** uvicorn не запущен):

   ```powershell
   python -c "from app.db.database import init_db, DB_PATH; init_db(); print(DB_PATH.resolve())"
   ```

3. **Проверка наличия файла БД:**

   ```powershell
   Test-Path ".\data\lead_discovery_crm_mvp.db"
   ```

   Ожидается `True`.

4. Зафиксировать в журнале прогона: дата/время, версия Python (`python --version`), полный путь, выведенный `DB_PATH`.

## Запуск двух инстансов

### Instance A (порт 8000)

В **терминале 1** из `<REPO>` с активированным venv:

```powershell
python -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

Базовый URL: `http://127.0.0.1:8000`

### Instance B (порт 8001)

В **терминале 2** из `<REPO>` с тем же venv и **тем же** `<REPO>`:

```powershell
python -m uvicorn app.main:app --host 127.0.0.1 --port 8001
```

Базовый URL: `http://127.0.0.1:8001`

### Важно

- Оба процесса используют **один и тот же** файл `data\lead_discovery_crm_mvp.db`.
- Процессы **независимы** (разные PID, разные порты); общий только файл БД на диске.
- Не запускать третий uvicorn на ту же БД в рамках этого runbook (scope OP-004 — два инстанса).

| Переменная | Как получить |
|------------|----------------|

## Test сценарий (пошагово)

Все шаги через **браузер** (проще воспроизвести без curl). После каждого шага — **проверка** (см. следующий подраздел).

### Шаг 1 — Instance **A**: создать lead

1. Открыть `http://127.0.0.1:8000/leads`
2. Заполнить форму «Создать Lead» (обязательные поля):
   - **Platform:** `multi_instance_test`
   - **Profile name:** `runbook_t01`
   - **Profile URL:** `https://example.com/p/runbook-t01` (валидный URL)
   - Остальные поля — по желанию или пусто
3. Нажать **Создать Lead**
4. Дождаться редиректа на карточку. Записать **`LEAD_ID`** из URL: `http://127.0.0.1:8000/leads/<LEAD_ID>`

**Проверка после шага 1:** на карточке в A статус **`new`**. В терминалах A/B **нет** строк с `database is locked` / `database table is locked`.

### Шаг 2 — Instance **B**: обновить статус лида

1. Открыть `http://127.0.0.1:8001/leads/<LEAD_ID>` (подставить записанный id)
2. В блоке «Обновление статуса (FSM)» выбрать **`reviewed`** (единственный допустимый переход из `new`)
3. Нажать **Обновить статус**

**Проверка после шага 2:** на B в карточке статус **`reviewed`**. В A обновить страницу `http://127.0.0.1:8000/leads/<LEAD_ID>` (F5) — статус тоже **`reviewed`**. Логи обоих процессов без необработанных SQLite lock ошибок.

### Шаг 3 — Instance **A**: обновить notes

1. На `http://127.0.0.1:8000/leads/<LEAD_ID>` в блоке «Заметки» ввести текст, например: `notes_from_instance_A`
2. Нажать **Сохранить заметку**

**Проверка после шага 3:** на A в профиле видны новые заметки. На B открыть/обновить `http://127.0.0.1:8001/leads/<LEAD_ID>` — тот же текст заметок.

### Шаг 4 — Instance **B**: создать contact attempt

1. На `http://127.0.0.1:8001/leads/<LEAD_ID>` в «Contact Log» заполнить, например:
   - **Текст сообщения:** `msg_from_B`
   - **Outcome:** `sent`
   - **Next action:** `wait_reply`
   - Дату можно оставить пустой (подставится текущая на стороне сервера)
2. Нажать **Добавить попытку контакта**

**Проверка после шага 4:** в «Истории контактов» на B видна новая строка. На A обновить карточку — **та же** строка (тот же контент).

### Шаг 5 — Instance **A**: создать consultation

1. На `http://127.0.0.1:8000/leads/<LEAD_ID>` в «Consultation Management»:
   - **Planned at:** `2026-06-01 10:00:00` (формат как в placeholder шаблона)
   - **Status:** `planned`
   - **Result:** можно пусто
2. Нажать **Создать консультацию**
3. Записать **`CONSULTATION_ID`** из таблицы «Список консультаций» (колонка **ID** новой строки)

**Проверка после шага 5:** консультация видна на A. На B обновить карточку — та же консультация с тем же `CONSULTATION_ID`. Логи без необработанных lock ошибок.

### Шаг 6 — Instance **B**: обновить consultation (status / result)

1. На `http://127.0.0.1:8001/leads/<LEAD_ID>` в строке консультации с id = **`CONSULTATION_ID`**:
   - **Status:** выбрать **`completed`**
   - **Result:** например `result_from_B`
2. Нажать **Сохранить** в этой строке

**Проверка после шага 6:** на B в таблице у этой консультации статус **`completed`**, результат обновлён. На A обновить страницу — **те же** статус и результат.

## Проверки (контрольный чеклист)

| # | Что проверить | Как |
|---|----------------|-----|
| P1 | Нет необработанных ошибок SQLite | В обоих окнах uvicorn искать traceback или сообщения `database is locked` / `database table is locked` **без** перехвата приложением (ответ 500 в браузере при нормальных действиях — тоже сигнал) |
| P2 | Данные консистентны | После шагов с записью: сравнить карточку `LEAD_ID` на A и B после **полного обновления** страницы (F5) |
| P3 | Изменения видны между инстансами | Каждый шаг 2–6: явное обновление «другого» порта и совпадение полей |
| P4 | Поведение retry (transient) | Если в логах **нет** ошибок блокировки — считать, что в данном прогоне коллизий не было или retry отработал прозрачно; при появлении ошибки — зафиксировать в отчёте VAL (не игнорировать) |

## Expected result

- Оба инстанса остаются запущенными без падений на протяжении сценария.
- Не наблюдается **потеря** шагов (каждое действие отражается в БД и видно с обоих портов после refresh).
- При кратковременных блокировках допустимо кратковременное ожидание за счёт `busy_timeout` и `run_write_with_retry` на write path; **итог** — успешные ответы и согласованные данные.

---

## Failure criteria (остановить прогон и зафиксировать)

- В логах uvicorn или в UI: необработанные **`database is locked`** / **`database table is locked`** (после исчерпания retry), необработанные **500** на шагах сценария без объяснения.
- **Потеря данных:** действие выполнено (успешный редирект), но после refresh данные не появились ни на A, ни на B.
- **Несогласованность:** после refresh на A и B расходятся обязательные поля (статус лида, заметки, контакт, консультация) для одного и того же `LEAD_ID` / `CONSULTATION_ID`.

При любом failure: сохранить скриншоты/логи, записать точное время и шаг; не менять код в рамках T01 — передать в материалы для [DOA-VAL-003](../validation_reports/DOA-VAL-003.md) (когда будет создан по OP-004).
