## Scope

- Валидация **двухинстансной приёмки** SQLite для MVP: два процесса `uvicorn` на `127.0.0.1:8000` и `127.0.0.1:8001`, **один** файл БД по `DB_PATH`.
- Опора на воспроизводимую процедуру [DOA-RUN-001](../runbooks/DOA-RUN-001.md) и фактический прогон [DOA-EXEC-001](../execution_logs/DOA-EXEC-001.md) (T01–T02 цикла [DOA-OP-004](../operational_plan/DOA-OP-004.md)).
- Проверяется **поведение приложения** при чередовании write/read операций между инстансами на общей БД, в границах сценария шагов 1–6 runbook (lead → status → notes → contact → consultation → update consultation).

Не входит в scope: stress/load, более двух инстансов, изменение кода или UI.

## Validation questions

| Вопрос | Ответ |
|--------|--------|
| Удалось ли воспроизвести двухинстансный сценарий по существу runbook? | **Да.** Оба порта, общий `DB_PATH`; шаги 1–6 выполнены с чередованием A/B согласно [DOA-EXEC-001](../execution_logs/DOA-EXEC-001.md). Метод исполнения — HTTP-эквивалент SSR-форм (заявлено в EXEC), а не ручной браузер — см. Known limitations. |
| Видны ли изменения между инстансами? | **Да.** Операции на A/B применялись к одному `LEAD_ID=12`; итог проверен через GET и согласованность полей в БД (EXEC). |
| Есть ли признаки SQLite lock failure (необработанные `database is locked` / `database table is locked`)? | **Не зафиксировано.** В EXEC: нет таких сообщений в наблюдаемых HTTP-ответах и проверке БД; полных логов uvicorn в файл не было — см. limitations. |
| Согласованы ли данные после сценария? | **Да.** EXEC фиксирует: lead `reviewed` с заметками, contact с `msg_from_B`, consultation `id=6` в статусе `completed` с `result_from_B`. |
| Влияет ли дефект harness (regex/HTML parsing на шаге 6) на **product verdict**? | **Нет.** В EXEC явно разделено: сбой извлечения `CONSULTATION_ID` из HTML — ошибка **вспомогательного** скрипта; идентификатор подтверждён через SQLite, POST шага 6 на инстансе B выполнен успешно. Это **не** дефект приложения и **не** индикатор поломки SQLite runtime. Влияет только на оценку **полноты автоматизации** прогона (execution method limitation). |

## Result summary

- Шаги runbook **1–6** по смыслу и порядку инстансов **выполнены** ([DOA-EXEC-001](../execution_logs/DOA-EXEC-001.md)).
- Операции **lead / contact / consultation** завершились **успешно** (HTTP 200, редиректы SSR, согласованное состояние в БД).
- Данные после сценария **согласованы** с ожидаемым сценарием.
- **Не** зафиксировано необработанных ошибок блокировки SQLite на наблюдаемом пути исполнения.
- **Intermediate verdict `partial` в DOA-EXEC-001** относится к **строго автоматизированному** проходу шага 6 без доразметки id и к **методу сбора доказательств**, а не к корректности продукта или SQLite hardening.
- Итоговая оценка в рамках **OP-004** касается **продуктового** поведения при двух инстансах на одной БД: в объёме данного прогона оно **соответствует** цели приёмки.

## Known limitations

- Вывод **uvicorn** (stdout/stderr) **не** сохранялся в файл; отсутствие lock-ошибок подтверждено косвенно (успешные ответы, консистентность БД), а не полным текстовым логом сервера.
- Прогон **не** является stress-/load-тестом и **не** доказывает поведение при большем числе параллельных писателей (согласовано с духом [DOA-VAL-002](./DOA-VAL-002.md)).
- Исполнение было **HTTP-эквивалентом** SSR flow, а не ручным прохождением в браузере как в тексте [DOA-RUN-001](../runbooks/DOA-RUN-001.md); семантика запросов совпадает с формами приложения.
- На шаге 6 **`CONSULTATION_ID`** для финального POST был подтверждён через **прямой запрос к SQLite**, поскольку harness не извлёк id из HTML — ограничение **метода исполнения**, зафиксированное в EXEC и здесь для прозрачности повторяемости.

## Verdict

**passed with limitations**

**Кратко:** двухинстансную приёмку в объёме **OP-004** и фактического прогона **T01–T02** считать **успешной для продукта и SQLite runtime в наблюдаемых условиях**: сценарий воспроизведён, данные согласованы, признаков необработанных SQLite lock failures не зафиксировано. Оговорки (**limitations**) относятся к полноте **доказательной базы** (логи сервера, не браузер, не нагрузочный тест) и к **исполнительскому harness** на шаге 6 — без переноса этих пунктов на дефект приложения.
