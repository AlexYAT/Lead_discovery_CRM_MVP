## Title

Dotenv-based Config Loading Architecture

---

## Overview

**DOA-IDEA-012** фиксирует пробел: в **локальном** runtime файл **`.env`** не гарантируется загруженным до чтения настроек, из‑за чего типичный запуск зависит от shell.

Архитектурно вводится **централизованная** стадия **инициализации окружения** перед использованием существующего **Config Layer** для discovery (и при необходимости — других entrypoints в рамках отдельного DEC), **без** изменения бизнес-логики **pipeline** и **без** пересборки **env contract** (имена переменных и семантика остаются baseline до отдельного решения).

Выбор конкретной библиотеки (**python-dotenv** и аналоги) **не** фиксируется в ARCH — переносится на **DEC/OP/IMP**.

---

## Architectural Approach

Вводится этап:

### Environment Initialization Layer

**Положение в жизненном цикле процесса:**

**Application Start → Environment Initialization → Config Layer → Application Components**

**Назначение:** один раз (идемпотентно, по правилам следующего DEC) подготовить **процессное окружение** для локального запуска из корня репозитория, чтобы последующий **`load_config_from_env()`** и аналоги видели те же переменные, что описаны в **`.env.example`** / **README**, без требования ручного `export` в каждой сессии.

---

## Components

### Environment Initialization

- отвечает за **чтение** файла **`.env`** (если файл существует) и **запись** значений в **процессное окружение** (`os.environ` / эквивалент), согласованно с политикой «не перезатирать уже заданные переменные окружения» (**override policy** — в **DEC**);  
- выполняется **до** первого обращения приложения к **Config Layer** для чтения discovery-настроек;  
- **не** является поставщиком typed-конфигурации для модулей: **не** экспортирует значения «напрямую» в бизнес-код, только подготавливает **env**, из которого читает **Config Layer**.

### Config Layer (existing)

- остаётся **единственной** точкой **чтения** настроек discovery из **уже инициализированного** `os.environ` (**строгая** модель для ключей и `DISCOVERY_*`, как в текущем baseline);  
- **не** читает файл **`.env`** самостоятельно;  
- контракт полей **`DiscoveryConfig`** расширяется **только** через отдельные **DEC/IMP**, не через инициализационный слой.

---

## Data Flow

**.env** (локальный файл, не в VCS) → **Environment Initialization** (чтение файла, запись в процессное окружение по правилам) → **`os.environ`** → **Config Layer** (`load_config_from_env` / merge CLI) → **компоненты приложения** (orchestration, discovery pipeline).

---

## Constraints

- **Запрещено** читать файл **`.env`** напрямую из произвольных модулей приложения вне **Environment Initialization** (одна точка загрузки файла).  
- **Запрещено** использовать **`os.getenv` / `os.environ`** в discovery-модулях **вне** **`app/discovery/config.py`** для настроек, уже покрытых **config discipline** (baseline сохраняется).  
- **Не** менять бизнес-логику **pipeline** discovery в рамках этого архитектурного цикла.  
- **Не** менять существующий **env contract** (перечень переменных и пользовательская документация) **сверх** необходимости описать точку вызова инициализации в **README** — детали в **OP/IMP**.
