# Summary

Архитектурная фиксация слоя **candidate queue (draft)** и явного перехода **candidate → lead** без смешения с существующей моделью `leads` и FSM CRM. Основано на [DOA-IDEA-004](../ideas/DOA-IDEA-004.md) (curated inbound + очередь на разбор).

---

## Scope

| Входит | Не входит (см. Non-goals) |
|--------|---------------------------|
| Модель **candidate** как черновик до CRM | Автопоиск, скрейпинг, enrichment |
| **Импорт** кандидатов (CSV и/или paste — детали в DEC) | Изменение текущей FSM lead |
| **Convert to lead** как единственный вход новых `leads` из очереди (в рамках MVP-итерации — по решению DEC) | Multi-user, внешние OAuth-провайдеры |
| Отдельные **UI/API границы** для очереди vs карточка лида | Объединение таблиц `candidates` и `leads` |

## Core model

### Candidate (draft)

Логическая сущность черновика **до** принятия в CRM. Минимальный набор полей (расширение — только через явный scope):

| Поле | Назначение |
|------|------------|
| **id** | Суррогатный ключ черновика |
| **platform** | Канал/платформа источника (строка, согласуемая с семантикой lead) |
| **profile_name** | Отображаемое имя / ник |
| **profile_url** | Стабильная ссылка на профиль |
| **notes** | Свободный текст оператора (контекст до конверсии) |
| **status** | Жизненный цикл **только** черновика: `new` → `accepted` \| `rejected` (и терминальные состояния; точные переходы — в DEC) |

**Инвариант:** `candidate.status` **не** является FSM статусом лида (`new`, `reviewed`, …). Это отдельный мини-конечный автомат очереди.

### Relationship

- **Candidate ≠ Lead.** Разные сущности, разные идентификаторы; запрещено трактовать строку очереди как lead «в статусе draft» внутри таблицы `leads`.
- Переход в CRM — только через явное действие **«convert to lead»** (имя в UI/API может варьироваться, семантика фиксирована).
- После успешной конверсии:
  - создаётся строка в **`leads`** с переносом согласованного набора полей (минимум: platform, profile_name, profile_url, notes — маппинг в DEC);
  - у candidate фиксируется связь с результатом: **`lead_id`** (nullable до конверсии), чтобы аудировать происхождение лида;
  - опционально (DEC): **`candidate_id`** на стороне `leads` для обратной трассировки **или** отдельная таблица связи — без дублирования бизнес-данных в двух местах.

**Инвариант:** одна успешная конверсия → один новый `lead_id`; повторный convert того же candidate **запрещён** на уровне правил (DEC: unique constraint / статус `accepted`).

## Flow

1. **Импорт кандидатов** — загрузка CSV и/или вставка списка (формат и парсер — DEC); создание записей **только** в слое candidates со статусом `new`.
2. **Отображение в очереди** — отдельный экран/маршрут «Candidates»: список, сортировка по дате создания, фильтр по status (минимум).
3. **Ручная фильтрация** — пользователь помечает строки `rejected` или оставляет `new` для разбора; редактирование notes до конверсии (DEC).
4. **Convert → lead** — для выбранного `accepted` path: вызов доменной операции «создать lead из candidate», фиксация `lead_id` на candidate, перевод candidate в терминальный статус очереди (например `accepted` как «принят в CRM» — уточнить именование в DEC, чтобы не путать с lead FSM).
5. **Дальнейшая работа** — только существующий CRM flow: `/leads/{id}`, контакты, консультации, dashboard; очередь не подменяет карточку лида.

## API / UI boundaries

| Граница | Правило |
|---------|---------|
| Данные | Таблица(ы) **candidates** отдельно от **`leads`**; не объединять в одну таблицу с полиморфным «типом строки». |
| Маршруты | Префикс вида `/candidates` (или эквивалент) отдельно от `/leads`; не встраивать черновики в список лидов как «ещё один фильтр». |
| UI | Минимум: список очереди + действия (reject / convert), детальная карточка candidate **опционально** в MVP-итерации (DEC). |
| API (если появится JSON-слой) | Namespace `/api/candidates` vs существующие SSR POST — не смешивать обработчики convert с `POST /leads` без явного различия семантики. |

## Non-goals

- Автоматический **поиск** кандидатов во внешних сетях.
- **Scraping** и обход ToS площадок.
- **Enrichment** (автоподтягивание метаданных по URL) на данном архитектурном этапе.
- Унификация candidate и lead в одной таблице ради «простоты схемы».

## Risks

| Риск | Контроль (на уровне архитектуры) |
|------|----------------------------------|
| **Дубли** (тот же profile_url и в candidates, и в leads) | Уникальность/предупреждение на конверсии (DEC); опционально нормализация URL перед сравнением. |
| **UX перегрузка** | Два чётких режима: «очередь» vs «CRM»; не дублировать консультации/контакты на слое candidate. |
| **Хранение лишних данных** | Отклонённые кандидаты: политика retention / soft-delete (DEC); не раздувать поля candidate без IDEA. |

## Decisions to follow (DEC placeholders)

Следующий **DOA-DEC-xxx** должен зафиксировать:

1. **Хранение candidates** — отдельная таблица `candidates` (рекомендуемый дефолт в ARCH) vs альтернатива; схема колонок включая `lead_id`, timestamps.
2. **Soft delete vs status** — удаление физическое vs статус `rejected` + опциональное скрытие из списка по умолчанию.
3. **Массовая конверсия** — разрешён ли batch convert из CSV-импорта в MVP-итерации или только одиночный convert.
4. **Именование терминального статуса** после convert (`accepted` на candidate vs `converted`) — избежать коллизии с термином «accepted» в других доменах.

Реализация (миграции, сервисы, UI) **не** входит в объём настоящего документа.
