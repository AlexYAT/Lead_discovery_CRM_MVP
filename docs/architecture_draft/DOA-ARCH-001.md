# ARCH: Lead Discovery CRM - MVP Architecture Draft

## Summary
Документ переводит идею `DOA-IDEA-001` в архитектурный черновик MVP-системы Lead Discovery CRM.
Фокус - зафиксировать архитектурную рамку, модель данных и базовые технические решения без детализации
реализации и без перехода к этапам DEC/OP/IMP.

## Goal
Архитектурная цель - превратить концепцию ручного lead discovery в работающую систему, где:
- данные о лидах хранятся централизованно;
- ручные контакты и их исходы фиксируются в истории;
- статусная модель воронки поддерживается на уровне данных и интерфейса;
- ключевые метрики доступны в dashboard для контроля прогресса MVP.

## Scope
### Входит в архитектурный контур MVP
- backend-сервис на FastAPI;
- локальное хранилище SQLite;
- SSR UI для работы пользователя с CRM;
- доменная модель: Lead, Contact Attempt, Consultation;
- базовые модули: Lead CRM, Contact Log, Dashboard.

### Исключено из архитектурного контура MVP
- автоматическая рассылка/автоконтакт;
- AI-слой продаж и авто-рекомендации;
- сложные внешние интеграции и фоновые коннекторы источников.

## 1. Архитектурная цель
Идея из `DOA-IDEA-001` трансформируется в архитектуру как последовательность:
1. Фиксация сущностей и их связей.
2. Определение минимального backend-контракта между UI и данными.
3. Поддержка ручного операционного процесса через UI и журнал действий.
4. Формирование наблюдаемой воронки на основе статусной модели и агрегированных метрик.

## 2. Общая архитектура
Архитектура MVP состоит из трех слоев:
- **FastAPI backend**: бизнес-логика, валидация входных данных, управление статусами и журналом контактов.
- **SQLite storage**: хранение доменных сущностей, связей и истории.
- **SSR UI**: интерфейс работы с лидами, контактами, консультациями и dashboard.

Взаимодействие компонентов:
1. SSR UI отправляет действия пользователя в backend.
2. Backend валидирует команду, изменяет данные в SQLite и возвращает обновленное состояние.
3. UI отображает актуальный список/карточку/метрики на основе полученных данных.

## 3. Модель данных (архитектурный уровень)
### Lead
Сущность лида хранит первичную информацию о потенциальном клиенте, оценку релевантности и текущий статус
в воронке.

### Contact Attempt
Сущность попытки контакта хранит факт взаимодействия, текст сообщения, результат и следующее действие.
Связь: один `Lead` -> много `Contact Attempt`.

### Consultation
Сущность консультации хранит планирование и итог целевого действия после успешного контакта.
Связь: один `Lead` -> ноль или много `Consultation`.

### Связи между сущностями
- `Lead` является корневой сущностью.
- `Contact Attempt` и `Consultation` привязаны к `Lead` по `lead_id`.
- История коммуникации и консультаций формирует контекст для обновления статуса лида.

## 4. Основные модули системы
### Lead CRM
- список лидов;
- фильтрация по статусам и атрибутам;
- карточка лида;
- обновление статуса;
- заметки.

### Contact Log
- регистрация попытки контакта;
- фиксация исхода;
- установка следующего шага;
- просмотр истории по лидy.

### Dashboard
- агрегированный счетчик лидов;
- метрики по статусам;
- базовые conversion-метрики между этапами воронки.

## 5. Потоки данных
### Создание лида
UI -> backend -> создание `Lead` в SQLite -> возврат созданной сущности -> отображение в списке и карточке.

### Обновление статуса
UI отправляет новый статус -> backend проверяет допустимость перехода -> обновление `Lead.status` ->
синхронизация отображения списка/карточки/dashboard.

### Логирование контакта
UI фиксирует попытку -> backend создает `Contact Attempt` с привязкой к `lead_id` -> история обновляется ->
при необходимости пересчитывается текущий статус.

### Отображение dashboard
UI запрашивает агрегаты -> backend вычисляет метрики по данным SQLite -> UI показывает текущее состояние воронки.

## 6. Границы системы
### Что входит в MVP
- полный ручной цикл: создание лида -> контакт -> фиксация результата -> управление консультацией;
- хранение истории и статусов;
- базовая аналитика в dashboard.

### Что явно исключено
- автоматизированные сценарии outreach;
- интеллектуальные рекомендации и AI-обработка;
- интеграционный слой с внешними платформами в реальном времени.

## 7. Ограничения архитектуры
- single-user модель эксплуатации;
- SQLite как единственное хранилище MVP;
- отсутствие асинхронных интеграций и очередей;
- отсутствие AI-слоя;
- приоритет простоты и прозрачности над расширяемостью.

## 8. Технические решения (на уровне draft)
### Структура backend
- слой API (входной контракт UI -> backend);
- слой доменной логики (правила статусов, валидация операций);
- слой доступа к данным (операции чтения/записи в SQLite).

### Подход к хранению
- реляционная схема с явными связями через `lead_id`;
- хранение истории контактов как отдельных записей;
- сохранение данных на локальном уровне с устойчивостью к перезапуску приложения.

### Базовая схема API (без детализации endpoint'ов)
- операции управления лидами (создание, просмотр, обновление статуса, заметки);
- операции журнала контактов (добавление записи, просмотр истории);
- операции консультаций (планирование, обновление результата);
- операции dashboard (получение агрегированных метрик).

## 9. Риски
- ограниченная масштабируемость SQLite при росте объема данных и конкурентности;
- риск неполноты данных из-за ручного ввода;
- зависимость от дисциплины пользователя при фиксации всех шагов;
- отсутствие автоматизации может замедлять операционный цикл при росте потока лидов.
