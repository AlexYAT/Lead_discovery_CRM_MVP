# Summary
Архитектура проведения validation-цикла UX для MVP.

# Цель
Определить, как будет выполняться проверка системы пользователем в рамках validation-cycle без изменения
реализации системы.

# Подход к validation
- ручное прохождение сценариев;
- фиксирование наблюдений;
- отсутствие изменений в системе.

# Роли
- пользователь (исполнитель сценариев);
- наблюдатель (фиксирует проблемы).

# Сценарии (ref IDEA)
- создание лида;
- работа со списком;
- карточка лида;
- contact log;
- consultation;
- dashboard.

# Метод фиксации
- markdown-документы;
- единый формат записи проблем.

# Структура будущих документов
Предлагаемая структура:

`docs/validation_reports/`
- `DOA-VAL-XXX.md`

# Ограничения
- никаких изменений системы;
- никаких решений;
- только фиксация.

# Риски
- субъективность оценки;
- неполное покрытие сценариев.

# Definition of Done
- определён процесс validation;
- определена структура фиксации;
- готово к DEC.
